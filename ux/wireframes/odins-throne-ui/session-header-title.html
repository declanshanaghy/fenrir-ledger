Title source priority: (1) PR title from gh pr list --head <branch> if available, (2) Issue title + step number, (3) Branch-name parse fallback (e.g. "fix/issue-987-picker-gate" → "Issue #987 – picker gate – Step 1"). Backend resolves at job-creation time and stores as K8s annotation/label so no runtime API calls are needed.

4 · Title Construction — Priority Order

# Source Example output When used
1 PR title (GitHub API) Issue #987 – fix: gate /api/config/picker on Karl OR active trial – Step 1 A PR exists with the job branch as its head. Fetched at job-creation time, stored as K8s annotation fenrir/pr-title.
2 Issue title (GitHub API) Issue #987 – fix: gate /api/config/picker on Karl OR active trial – Step 1 No PR yet. Issue title fetched at job-creation time, stored as K8s annotation fenrir/issue-title.
3 Branch name parse (no API) Issue #987 – picker gate – Step 1 Fallback when API calls fail or are unavailable. Branch fix/issue-987-picker-gate parsed: type stripped, hyphens → spaces, issue extracted.
4 Raw job name issue-987-step1-firemandecko-e96269d4 Last resort — no branch, no API response. Displayed in header meta row only (not as primary title).

Recommendation: fetch title at dispatch time (in dispatch-job.sh or k8s.ts job-creation path) and store as K8s Job annotation. The monitor backend reads the annotation from the Job object, avoiding runtime GitHub API calls per subscription. See Implementation Notes below.

6 · DisplayJob Type — Proposed Extension

Field Type Source Notes
issueTitle string | null K8s annotation fenrir/issue-title or PR title annotation fenrir/pr-title The resolved human-readable title. Null triggers branch-parse fallback on the client.
branchName string | null K8s Job label fenrir/branch (already present in dispatch scripts) Used for branch-parse fallback when issueTitle is null.

Existing fields issue, step, agentName, sessionId are unchanged. The UI constructs the display title client-side using: issueTitle ?? parseBranchTitle(branchName) ?? rawJobName.

8 · Accessibility Requirements (WCAG 2.1 AA)

  • aria-label on content header: "Active session: Issue #987 – fix: gate /api/config/picker on Karl OR active trial – Step 1" — gives screen readers the full unstemmed title, not the truncated visual version.
  • Agent badge: aria-label="Agent: FiremanDecko"
  • Session ID element: aria-label="Session ID: issue-987-step1-firemandecko-e96269d4" (full ID, not truncated)
  • Sidebar job cards: existing aria-label={`Job: ${agentName} issue ${issue} step ${step}`} should be updated to aria-label={`Job: Issue ${issue} – ${issueTitle} – Step ${step} – ${agentName} – ${status}`} when issueTitle is available.
  • Tooltip (title attr) on session ID is supplemental only. The aria-label on the element already contains the full ID for assistive tech.
  • Minimum touch target 44×44px applies to the download button. The .wf-download-btn size of 28×28px visual area should be padded to meet this requirement.

9 · Implementation Notes for FiremanDecko

Backend — monitor/src/k8s.ts

  • Read K8s Job annotations fenrir/issue-title and fenrir/branch from the Job object in the job-listing query.
  • Include them in the Job wire type: issueTitle: string | null, branchName: string | null.
  • No runtime GitHub API calls needed — titles are resolved at dispatch time.

Backend — dispatch/dispatch-job.sh (or equivalent)

  • At job creation time, run gh issue view $ISSUE --json title -q .title and store as annotation fenrir/issue-title.
  • Also try gh pr list --head $BRANCH --json title -q .[0].title and store as annotation fenrir/pr-title (may be empty if no PR yet).
  • Prefer fenrir/pr-title over fenrir/issue-title when both are present (PR title is more specific).

Frontend — monitor-ui/src/lib/types.ts

  • Add issueTitle: string | null and branchName: string | null to DisplayJob.
  • Add a utility resolveSessionTitle(job: DisplayJob): string in lib/ that implements the priority order from Section 4.

Frontend — LogViewer.tsx (content-header)

  • Replace single <span className="session-title"> with the two-row title block from Section 1 "After".
  • Title block: <span className="session-title-primary"> (full resolved title) + meta row with agent badge, step tag, truncated session ID.
  • Truncation of session ID: last 8 chars + ellipsis prefix. Tooltip contains full session ID.
  • This pattern applies to all three render paths (normal, TTL-expired, node-unreachable) — extract a SessionHeader sub-component to avoid triple duplication.

Frontend — JobCard.tsx (sidebar)

  • Replace card-agent + card-meta + session ID div with the new two-row layout from Section 2 "After".
  • Issue title as primary text (line 1), truncated with overflow: hidden; text-overflow: ellipsis; white-space: nowrap.
  • Agent badge + step + status on line 2.
  • Session ID removed from card entirely.

Edge cases

  • No issueTitle and no branchName: show raw sessionId as the primary title (existing behavior, graceful degradation).
  • Very long issue titles (>80 chars): truncate in both header and card with ellipsis. Full title available in title tooltip on the primary title element.
  • Completed sessions: same layout — issueTitle was stored at job-creation time, so it's available even after the job ends.
  • Fixture/replay sessions: issueTitle comes from the fixture JSON. Fixture files should include issueTitle and branchName fields if present; fall back gracefully if missing.

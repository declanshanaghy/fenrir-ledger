<div className="epigraph">
The wolf does not howl without cause.

We built an autonomous agent orchestration system on top of <code>depot claude</code>.
Multiple agents — engineers, testers, designers — each running in their own sandbox,
each handing off to the next via git commits and GitHub issue comments.
The architecture is sound. The wolf runs.

But the chain that binds us is not the work itself — it is the tooling around it.
We have fought these battles long enough to know the difference between friction
that sharpens and friction that bleeds. This is the latter.

We are writing this because we *want* to stay. Depot's sandbox model is the
closest thing to what we need. But a wolf that cannot see its pack, cannot hear
when the hunt ends, and must claw through the underbrush for every scrap of
information — that wolf will eventually find another forest.

If these gaps remain, we will be forced to evaluate alternatives — raw EC2
sandboxes, modal.com, or rolling our own container orchestration. None of those
options are as elegant as what Depot *could* be. That's what makes this
worth writing.

What follows are the thorns we pulled from our paws. We offer them not as complaints,
but as prey laid at your door — problems worth hunting, before the wolf moves on.
</div>

<div className="rune-hr"><span>ᚠ ᛖ ᚾ</span></div>

<div className="grievance-card">
<span className="grievance-label">Grievance I</span>
<h2>No Simple Way to Retrieve a Session Log</h2>
<p className="grievance-summary">There is no <code>depot claude session-log &lt;id&gt;</code> command.</p>
<p className="grievance-body">To get the transcript of what an agent did in a sandbox, we must: search <code>~/.claude/projects/&lt;hash&gt;/</code> for JSONL files, filter by file size (Depot sessions are small; local conversations are large), grep inside each file for the session ID or branch name, and if not found locally, run <code>depot claude --wait --resume &lt;id&gt;</code> to trigger a download, then parse the JSONL with <code>jq</code>.</p>
<p className="grievance-body">We wrote a 90-line bash script to automate this. It should be one command.</p>

<span className="label-want">What we want</span>
<div className="code-block">
<pre>{`# Get the full session transcript
depot claude session-log <session-id>

# Raw JSONL for piping
depot claude session-log <session-id> --format jsonl

# Just assistant messages, human-readable
depot claude session-log <session-id> --format text`}</pre>
</div>

<span className="label-instead">What we have to do instead</span>
<div className="code-block">
<pre>{`# Our actual script — this is what "get a log" looks like today
JSONL_FILE=$(find ~/.claude/projects/ -name "*.jsonl" -size -500k \\
  -exec grep -l "$SESSION_ID" {} \\;)

if [ -z "$JSONL_FILE" ]; then
  depot claude --wait --resume "$SESSION_ID" --org "$ORG_ID" >/dev/null 2>&1
  JSONL_FILE=$(find ~/.claude/projects/ -name "*.jsonl" -size -500k \\
    -exec grep -l "$SESSION_ID" {} \\;)
fi

cat "$JSONL_FILE" | jq -r '...' # 40 more lines of jq incantations`}</pre>
</div>

<p className="grievance-closer">The wolf should not need to dig through filesystem internals to see what its pack did.</p>
</div>

<div className="grievance-card">
<span className="grievance-label">Grievance II</span>
<h2>CLI Defaults to Pager — Hostile to Automation</h2>
<p className="grievance-summary"><code>depot claude list-sessions</code> opens an interactive pager by default. This is death for automation.</p>
<p className="grievance-body">Every script that calls <code>list-sessions</code> must pass <code>--output json</code> to avoid hanging on a pager that will never receive a keypress. There is no environment variable (<code>DEPOT_PAGER</code>, <code>DEPOT_NO_PAGER</code>, <code>NO_PAGER</code>) to disable this globally.</p>
<p className="grievance-body">Even with <code>--output json</code>, the command prefixes a pagination token line before the JSON array, requiring an additional <code>sed -n '/^\[/,$ p'</code> to extract valid JSON. The pagination token should not be mixed into the JSON output — put it in stderr, a header, or a separate <code>--next-token</code> flag.</p>

<span className="label-want">What we want</span>
<div className="code-block">
<pre>{`# Environment variable to disable pager globally
export DEPOT_PAGER=off

# Or at minimum, detect non-TTY and disable automatically
depot claude list-sessions | jq '.[0].id'   # should just work`}</pre>
</div>

<span className="label-instead">What we have to do instead</span>
<div className="code-block">
<pre>{`depot claude list-sessions --output json 2>/dev/null \\
  | sed -n '/^\\[/,$ p' \\
  | jq '.[0].id'`}</pre>
</div>

<p className="grievance-closer">The wolf does not wait for permission to read its own rune log.</p>
</div>

<div className="grievance-card">
<span className="grievance-label">Grievance III</span>
<h2>Sessions Are Resumable — But We Want One-Shot</h2>
<p className="grievance-summary">The session model assumes you'll come back. We never do.</p>
<p className="grievance-body">The <code>depot claude</code> session model is built around resumption — start a session, leave, come back, resume. This is useful for interactive work. But our use case is <strong>fire-and-forget agent dispatch</strong>: launch an agent with a prompt, it does its work, it's done. We never resume it. We just want to know when it finishes.</p>
<p className="grievance-body">The <code>--wait</code> flag blocks until completion, which helps. But there's no lightweight way to check "is this session done yet?" without either running <code>--wait --resume</code> (blocks and downloads the full session) or listing all sessions and filtering (paginated, pager-hostile — see Grievance II).</p>

<span className="label-want">What we want</span>
<div className="code-block">
<pre>{`# Simple status check — returns immediately
depot claude session-status <session-id>
{ "id": "...", "status": "completed", "exit_code": 0, "duration": "3m42s" }

# Exit-code-based check for scripting
depot claude session-status <session-id> --quiet
# Exit 0 = completed, Exit 1 = running, Exit 2 = failed`}</pre>
</div>

<span className="label-instead">What we have to do instead</span>
<div className="code-block">
<pre>{`# Option A: Block forever
depot claude --wait --resume "$SESSION_ID" --org "$ORG_ID"

# Option B: Poll the session list (paginated, pager-hostile)
while true; do
  STATUS=$(depot claude list-sessions --output json 2>/dev/null \\
    | sed -n '/^\\[/,$ p' \\
    | jq -r ".[] | select(.id == \\"$SESSION_ID\\") | .status")
  [ "$STATUS" = "completed" ] && break
  sleep 30
done`}</pre>
</div>

<p className="grievance-closer">The wolf needs to watch all its pups at once, not sit beside each one until it sleeps.</p>
</div>

<div className="grievance-card">
<span className="grievance-label">Grievance IV</span>
<h2>No Backchannel for Session Completion</h2>
<p className="grievance-summary">When an agent finishes, nothing happens on our end. No webhook. No callback. No event.</p>
<p className="grievance-body">Our current architecture is "fire-and-forget, then the human checks manually." This works, but it means the human is the event loop. The wolf is leashed to the handler's schedule.</p>
<p className="grievance-body">Our best workaround is to run <code>depot claude --wait --resume</code> as a background process and rely on our CLI framework's task notification system. But this means holding an open process per session, which doesn't scale to 5+ parallel agent chains.</p>

<span className="label-want">What we want — webhooks</span>
<div className="code-block">
<pre>{`# Register a webhook for session lifecycle events
depot claude webhooks add \\
  --url "https://our-api.example.com/depot/callback" \\
  --secret "$WEBHOOK_SECRET" \\
  --events session.completed,session.failed

# Signed payload (HMAC-SHA256):
{
  "event": "session.completed",
  "session_id": "issue-199-step2-loki-fe3adaf1",
  "org_id": "pqtm7s538l",
  "exit_code": 0,
  "duration_seconds": 222,
  "repository": "declanshanaghy/fenrir-ledger",
  "branch": "fix/issue-199-cards-header-text"
}`}</pre>
</div>

<span className="label-want">Or even simpler — per-session callback</span>
<div className="code-block">
<pre>{`depot claude \\
  --session-id "issue-199-step2-loki" \\
  --on-complete "https://our-api.example.com/depot/done" \\
  -p "..."`}</pre>
</div>

<p className="grievance-closer">The wolf pack hunts in parallel. When one brings down its quarry, the others must know — without stopping to look over their shoulders.</p>
</div>

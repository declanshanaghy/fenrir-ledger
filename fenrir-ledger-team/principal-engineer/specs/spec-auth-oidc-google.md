## Database Schema

### Supabase PostgreSQL

All tables live in the `public` schema. UUIDs are generated by the application layer
(`crypto.randomUUID()`), not the DB, to maintain consistency with the TypeScript types.

#### Table: `households`

Maps 1:1 to the `Household` TypeScript interface, plus an `owner_google_sub` column that links
to the Google identity.

```sql
CREATE TABLE IF NOT EXISTS public.households (
    id              UUID        PRIMARY KEY,
    owner_google_sub TEXT       NOT NULL UNIQUE,  -- Google OIDC sub claim
    name            TEXT        NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ          DEFAULT NULL
);

-- Index for fast lookup by Google sub (used on every sign-in)
CREATE INDEX IF NOT EXISTS idx_households_owner_google_sub
    ON public.households (owner_google_sub);

-- RLS: enable
ALTER TABLE public.households ENABLE ROW LEVEL SECURITY;

-- RLS policy: users can only see their own household
-- The JWT claim 'household_id' is matched against the row id.
-- app.household_id is set by the API layer before executing queries.
CREATE POLICY "household_owner_isolation"
    ON public.households
    FOR ALL
    USING (id = current_setting('app.household_id', true)::UUID);
```

#### Table: `cards`

Maps 1:1 to the `Card` TypeScript interface. All money fields in cents (integer). All dates as
`TIMESTAMPTZ`. The `sign_up_bonus` field is a JSONB column matching the `SignUpBonus` interface.

```sql
CREATE TABLE IF NOT EXISTS public.cards (
    id                  UUID        PRIMARY KEY,
    household_id        UUID        NOT NULL REFERENCES public.households(id) ON DELETE CASCADE,
    issuer_id           TEXT        NOT NULL,
    card_name           TEXT        NOT NULL,
    open_date           TIMESTAMPTZ NOT NULL,
    credit_limit        INTEGER     NOT NULL DEFAULT 0,  -- cents
    annual_fee          INTEGER     NOT NULL DEFAULT 0,  -- cents
    annual_fee_date     TIMESTAMPTZ          DEFAULT NULL,
    promo_period_months INTEGER     NOT NULL DEFAULT 0,
    sign_up_bonus       JSONB                DEFAULT NULL,
    status              TEXT        NOT NULL CHECK (status IN ('active', 'fee_approaching', 'promo_expiring', 'closed')),
    notes               TEXT        NOT NULL DEFAULT '',
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ          DEFAULT NULL
);

-- Index for fast household-scoped card reads (most common query)
CREATE INDEX IF NOT EXISTS idx_cards_household_id
    ON public.cards (household_id)
    WHERE deleted_at IS NULL;

-- RLS: enable
ALTER TABLE public.cards ENABLE ROW LEVEL SECURITY;

-- RLS policy: users can only touch cards in their own household
CREATE POLICY "card_household_isolation"
    ON public.cards
    FOR ALL
    USING (household_id = current_setting('app.household_id', true)::UUID);
```

#### `sign_up_bonus` JSONB shape

Matches the `SignUpBonus` TypeScript interface:

```json
{
  "type": "points" | "miles" | "cashback",
  "amount": 60000,
  "spendRequirement": 400000,
  "deadline": "2025-06-15T00:00:00.000Z",
  "met": false
}
```

#### Soft-delete convention

- `deleted_at IS NULL` means active (same as TypeScript `deletedAt?: string` convention)
- All public API queries filter `WHERE deleted_at IS NULL`
- The `idx_cards_household_id` partial index covers only active cards for query performance

## Auth Middleware Strategy

### `development/src/middleware.ts`

Auth.js v5 exports an `auth` function that can be used directly as middleware.

```typescript
// Conceptual shape — implementation sprint only
export { auth as default } from "@/auth"

export const config = {
  matcher: [
    // Protect all routes except:
    // - /auth/signin and /auth/error (sign-in pages)
    // - /api/auth/* (Auth.js internal routes)
    // - /_next/* (Next.js internals)
    // - /static/* (marketing site)
    // - /favicon.ico, /robots.txt, /manifest.json
    "/((?!auth/signin|auth/error|api/auth|_next/static|_next/image|static|favicon.ico|robots.txt|manifest.json).*)",
  ],
}
```

The middleware checks for a valid JWT session. Unauthenticated requests to protected routes are
redirected to `/auth/signin?callbackUrl={encodedOriginalUrl}`. After sign-in, Auth.js redirects
to the original URL.

### Protected Route Group

`development/src/src/app/(protected)/layout.tsx` provides a server-side auth check as a
belt-and-suspenders layer on top of middleware. This handles edge cases where middleware is
bypassed (e.g., direct API calls in development).

```typescript
// Conceptual shape
import { auth } from "@/auth"
import { redirect } from "next/navigation"

export default async function ProtectedLayout({ children }) {
  const session = await auth()
  if (!session) redirect("/auth/signin")
  return <>{children}</>
}
```

## Session Shape

The Auth.js session object is extended to include `householdId`. This is stored in the JWT
payload and available in both server components and API routes without a DB lookup.

```typescript
// development/src/src/lib/session.ts

/** Shape of the Auth.js session available via auth() and useSession() */
export interface AppSession {
  user: {
    /** Google OIDC sub claim */
    sub: string
    /** User's full name from Google profile */
    name: string
    /** User's email from Google profile */
    email: string
    /** Google profile picture URL */
    image?: string
    /** Fenrir Ledger household UUID — set in Auth.js jwt() callback on first sign-in */
    householdId: string
  }
  expires: string
}
```

The `householdId` is injected into the JWT in the `jwt()` callback in `auth.ts`:
1. On first sign-in: look up or create the `Household` row in Supabase, then store the UUID.
2. On subsequent sign-ins: the `householdId` is already in the JWT token.

## Auth.js Configuration (`auth.ts`)

Conceptual shape:

```typescript
// development/src/auth.ts (conceptual — not final implementation)
import NextAuth from "next-auth"
import Google from "next-auth/providers/google"
import { SupabaseAdapter } from "@auth/supabase-adapter"
import { upsertHousehold } from "@/lib/db/households"

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    Google({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
  ],
  session: { strategy: "jwt" },
  callbacks: {
    async jwt({ token, account, profile }) {
      // On first sign-in (account is present), wire householdId into the token
      if (account && profile) {
        const household = await upsertHousehold({
          googleSub: token.sub!,
          displayName: profile.name ?? profile.email ?? "User",
        })
        token.householdId = household.id
      }
      return token
    },
    async session({ session, token }) {
      session.user.householdId = token.householdId as string
      session.user.sub = token.sub!
      return session
    },
  },
  pages: {
    signIn: "/auth/signin",
    error: "/auth/error",
  },
  trustHost: process.env.AUTH_TRUST_HOST === "1",
})
```

Note: The `SupabaseAdapter` is included for potential future use (if we add database sessions
later). In JWT mode it is not strictly required — the Supabase client is used directly in API
routes. The adapter can be omitted initially and added with a follow-up story if needed.

## API Routes

### `GET /api/cards` — List cards for authenticated household

```
Authorization: Session cookie (HttpOnly JWT)
Response 200: { cards: Card[] }
Response 401: { error: "Unauthorized" }
```

Handler pseudocode:
1. `const session = await auth()` — decode JWT
2. If no session, return 401
3. `getCardsByHousehold(session.user.householdId)` — DB query with `WHERE household_id = $1 AND deleted_at IS NULL`
4. Return cards

### `POST /api/cards` — Create a new card

```
Authorization: Session cookie
Request body: Omit<Card, 'id' | 'householdId' | 'createdAt' | 'updatedAt' | 'status'>
Response 201: { card: Card }
Response 400: { error: string }
Response 401: { error: "Unauthorized" }
```

Handler pseudocode:
1. Decode session, extract `householdId`
2. Validate request body with Zod schema
3. Generate `id = crypto.randomUUID()`
4. Compute `status = computeCardStatus(cardData)` (reuse existing `card-utils.ts`)
5. Insert row into `cards` table
6. Return created card

### `GET /api/cards/[id]` — Get single card

```
Authorization: Session cookie
Response 200: { card: Card }
Response 403: { error: "Forbidden" }  — card exists but belongs to different household
Response 404: { error: "Not found" }
Response 401: { error: "Unauthorized" }
```

Handler pseudocode:
1. Decode session
2. `getCardById(id)` — DB query
3. If `card.householdId !== session.user.householdId`, return 403 (never return 200 for wrong household)
4. Return card

### `PUT /api/cards/[id]` — Update a card

```
Authorization: Session cookie
Request body: Partial<Card> fields to update
Response 200: { card: Card }
Response 403: Forbidden
Response 404: Not found
```

### `DELETE /api/cards/[id]` — Soft-delete a card

```
Authorization: Session cookie
Response 204: No content
Response 403: Forbidden
Response 404: Not found
```

Handler: sets `deleted_at = NOW()` on the row. Reuses the same household isolation check as GET.

### `GET /api/households/me` — Get current user's household

```
Authorization: Session cookie
Response 200: { household: Household }
Response 401: Unauthorized
```

Returns the household record from Supabase matching `session.user.householdId`.

### Auth.js Internal Routes

`/api/auth/[...nextauth]` — handled entirely by Auth.js. This covers:
- `GET /api/auth/signin` — redirects to custom sign-in page
- `GET /api/auth/callback/google` — Google OAuth callback
- `POST /api/auth/signout` — session invalidation
- `GET /api/auth/session` — current session JSON (used by Auth.js client hooks)
- `GET /api/auth/csrf` — CSRF token

## Data Access Layer — `src/lib/db/`

### `client.ts` — Supabase client singleton

Creates a server-side Supabase client using the service role key for migration scripts and the
anon key for API routes. Sets the `app.household_id` Postgres configuration parameter for RLS.

```typescript
// Conceptual shape
import { createClient } from "@supabase/supabase-js"

/** Server-side Supabase client (anon key — RLS enforced) */
export function createSupabaseClient() {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}

/** Set the household_id for RLS enforcement on this connection */
export async function setHouseholdContext(
  supabase: ReturnType<typeof createSupabaseClient>,
  householdId: string
): Promise<void> {
  await supabase.rpc("set_config", {
    setting: "app.household_id",
    value: householdId,
    is_local: true,
  })
}
```

### `households.ts` — Household DB operations

```typescript
// Public interface (conceptual)
export async function upsertHousehold(params: {
  googleSub: string
  displayName: string
}): Promise<Household>

export async function getHouseholdById(id: string): Promise<Household | null>
```

`upsertHousehold` is the key function: called in the Auth.js `jwt()` callback on every sign-in.
Uses an `INSERT ... ON CONFLICT (owner_google_sub) DO NOTHING RETURNING *` pattern (or a
SELECT + INSERT if the row doesn't exist). This is the idempotent "create on first sign-in"
mechanism that retires `"default-household"`.

### `cards.ts` — Card DB operations

```typescript
// Public interface (conceptual)
export async function getCardsByHousehold(householdId: string): Promise<Card[]>
export async function getCardById(id: string, householdId: string): Promise<Card | null>
export async function insertCard(card: Omit<Card, "createdAt" | "updatedAt">): Promise<Card>
export async function updateCard(id: string, householdId: string, updates: Partial<Card>): Promise<Card>
export async function softDeleteCard(id: string, householdId: string): Promise<void>
```

Note: `householdId` is always a required parameter — this is the API-layer isolation enforcement.
The function never queries without a `WHERE household_id = ?` clause.

## TypeScript Strict Mode Notes

Auth.js v5 requires extending the `Session` and `JWT` types via module augmentation. Add to a
`types/next-auth.d.ts` file in `development/src/`:

```typescript
// development/src/types/next-auth.d.ts
import type { DefaultSession, DefaultJWT } from "next-auth"

declare module "next-auth" {
  interface Session {
    user: DefaultSession["user"] & {
      sub: string
      householdId: string
    }
  }
}

declare module "next-auth/jwt" {
  interface JWT extends DefaultJWT {
    householdId?: string
  }
}
```

This satisfies strict mode — `session.user.householdId` is typed without casting.

## Open Questions Surfaced by Engineering

1. **Luna sign-in page design**: The sign-in page visual spec is not yet written. This must be
   completed before the sign-in page story is marked "Ready". Copy direction is in Freya's story;
   visual layout (logo placement, button styling, background treatment) needs a Luna wireframe.

2. **Supabase project naming convention**: Should there be separate Supabase projects for
   development, staging (preview), and production? Or a single project with environment-scoped
   data? Recommendation: single production project + a separate development project. Preview
   deployments can share the dev project.

3. **`AUTH_TRUST_HOST` security review**: Confirm with Freya that preview deployments (which use
   dynamic Vercel URLs) are acceptable for Google OAuth. Google requires redirect URIs to be
   pre-registered — dynamic URLs cannot be individually pre-registered. Options: (a) use a
   wildcard domain match (not supported by Google), (b) use a fixed preview subdomain on a custom
   domain (complex), (c) accept that preview deployments can't initiate Google OAuth (just
   integration/UI tests) and only production uses real OIDC. This needs a decision before Sprint 3
   starts.

4. **RLS `set_config` approach**: Supabase RLS with `current_setting('app.household_id')` requires
   calling `set_config` on each connection before executing queries. Supabase's connection pooling
   (PgBouncer in transaction mode) resets settings between transactions. This means the
   `set_config(..., is_local=true)` approach must be used, which scopes the setting to the
   current transaction. All DB calls must be wrapped in a transaction or use the row-level
   approach. If this proves fragile, the fallback is API-layer-only isolation (remove RLS
   policies, keep all `WHERE household_id = ?` clauses). Evaluate during Sprint 3.

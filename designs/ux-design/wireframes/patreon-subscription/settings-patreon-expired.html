Scenario 3: Info Toast — Membership Expired

When the server detects an expired membership (on login or periodic check), a one-time info toast appears. It is shown once per session -- not on every page load. The toast uses the standard sonner toast component. Position: top-right on desktop, top-center on mobile. Duration: 6 seconds auto-dismiss, or manual dismiss via close button. This is NOT a blocking modal. The user can continue using the app.

Your Karl membership has expired. Premium features are locked.
Position: top-right (desktop) / top-center (mobile). Z-index: 300 (toast layer). Auto-dismiss after 6 seconds. Not repeated within the same session.
Expiration Detection Flow

The expiration is detected server-side, not client-side. Flow: 1. User signs in (Google OAuth). 2. Server checks Patreon membership status via Patreon API. 3. If membership was active but is now inactive/lapsed: a. Server updates the cached tier to "expired". b. The localStorage tier cache is updated on next page load. c. The TopBar badge changes from KARL to EXPIRED. d. The info toast fires once per session. 4. Premium features are locked immediately. 5. Card data is never affected by tier changes. Timing: - Checked on every Google sign-in (login event). - Optionally checked on a periodic interval (engineering decision). - The Patreon billing cycle is trusted -- we do not try to predict when a membership will lapse. We only detect the current status. Renewal: - When the user renews on Patreon (external action), the next login check will detect the active membership. - The EXPIRED badge changes back to KARL. - Premium features are unlocked immediately. - A success toast appears: "Welcome back, Karl."
